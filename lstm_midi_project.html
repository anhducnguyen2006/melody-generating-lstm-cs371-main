Project overview

What the team is building

Core idea

Train an LSTM on melody tokens extracted from MIDI piano music, then use a short seed melody to generate new token sequences and export them back to MIDI files.

Notebook scope

  • Dataset filtering from MAESTRO for Frédéric Chopin.
  • Melody extraction from full piano scores.
  • Tokenization, LSTM training, and MIDI generation.
  • Testing with held-out seed files from the test split.

Model concept

What is LSTM?

Simple definition

LSTM stands for Long Short-Term Memory. It is a recurrent neural network architecture built for sequential data, so it can learn from what happened earlier in a sequence.

In music generation, this helps because melody depends on earlier notes, rests, rhythm, and phrase direction rather than isolated events.

Why this model fits

  • Captures time-dependent musical patterns.
  • Uses prior context to predict the next token.
  • Handles fixed windows of 64 melody steps in this project.
  • Works well for seed-based continuation of MIDI melodies.

End-to-end pipeline

System workflow

1

Filter source data

Read MAESTRO metadata, keep Chopin files only, and copy them into train, validation, and test folders.

2

Extract melody

Flatten each MIDI score and select one melodic note per onset while preserving tempo, time signature, key, and rests.

3

Tokenize sequence

Encode notes as MIDI pitches, rests as r, and sustained durations as continuation token _.

4

Train LSTM

Use sliding windows of length 64, one-hot inputs, one LSTM layer with 256 units, and dropout.

5

Generate continuation

Prompt the model with a short seed melody, sample the next tokens with temperature control, and stop on a delimiter token.

6

Write MIDI output

Merge note and continuation tokens back into notes and rests, then save generated files in the results directory.

Feature engineering

Melody extraction logic

Musical assumptions

  • Melody is usually in the right hand, so notes at or above C4 are preferred.
  • If multiple notes start together, the algorithm selects one representative melodic pitch.
  • Pitch continuity matters, so large jumps are penalized.
  • Rests are inserted to preserve phrasing between note events.

Context preserved

  • Tempo marks are copied into the new melody track.
  • Time signatures remain available for rhythmic structure.
  • Key signatures keep the tonal context of the original piece.
  • Velocity is retained so dynamics are not completely lost.

Sequence representation

How MIDI becomes LSTM tokens

Token rules

60, 62, 64... note-on events as MIDI pitch values
r rest token
_ continuation of the current note or rest
/ delimiter repeated 64 times between files

Time resolution

Each token step uses TIME_STEP = 0.25, so duration is discretized into quarter-beat steps. The model sees fixed windows of 64 previous tokens and predicts the very next token.

Retrospective

Challenges and future improvements

⚠️

What we ran into

The model does not run well on the Chopin subset yet: outputs can become repetitive, unstable, or lose coherent phrasing.

🧪

Likely causes

Limited and stylistically complex data, heuristic melody extraction, and a simple single-layer LSTM setup constrain expressive modeling.

🚀

Recommendations

Expand and clean data, improve melody/chord representation, tune sequence length and temperature, and compare against Transformer-based baselines.
